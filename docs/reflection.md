# Widget Reflection

Every Bloom widget is described by a **reflection**.

A reflection is the canonical representation of a widget. It is produced by the build system and consumed by every code generator.

The runtime itself never performs reflection.

Everything generated by Bloom originates from the widget reflection.

---

## Philosophy

A widget has two responsibilities.

- Describe its state.
- Describe its interface.

State is represented by **properties**.

Interface is represented by **commands** and **events**.

Everything else is metadata.

---

## Runtime IDs

Bloom defines two completely different identifiers.

### Runtime ID

The runtime ID uniquely identifies a widget instance.

It is assigned when the widget is created.

The runtime ID is used for:

- Read
- Update
- Delete
- Event dispatch
- Serialization

Every widget instance has a unique runtime ID.

---

### Constructor ID

The constructor ID uniquely identifies a widget type.

It is assigned at build time.

The constructor ID is used for:

- Widget creation
- Serialization
- Restoration
- Host construction

Every widget class has exactly one constructor ID.

---

## Widget Reflection

A widget reflection consists of four parts.

```text
Widget Reflection

Metadata
Props
Interface
Hierarchy
```

---

## Metadata

Metadata describes the widget itself.

Metadata never changes at runtime.

Examples include:

- Constructor ID
- Widget name
- Display name
- Icon
- Category
- Description

Metadata is intended for tooling.

Examples:

- Widget palette
- Documentation
- Inspector
- Search

---

## Props

Props define the complete state of a widget.

At any point in time, the properties completely describe how a widget exists.

Properties include things such as:

- Text
- Position
- Size
- Color
- Visibility
- Selection

Properties are reactive.

The host may modify properties.

The runtime synchronizes property changes automatically.

Properties are also the serialization format of a widget.

---

## Interface

The interface defines everything a widget can do.

An interface consists of:

### Commands

Commands are actions invoked by the host.

Examples:

- Focus
- Blur
- Open
- Close
- ScrollTo

Every command is synchronized through the transport layer.

Commands generate strongly typed wrappers for every supported language.

---

### Events

Events are actions emitted by the runtime.

Examples:

- Click
- Hover
- Changed
- Closed

Events are dispatched through the transport layer.

Hosts subscribe to events using generated bindings.

---

## Hierarchy

Hierarchy describes how widgets compose.

Examples include:

- Accepts children
- Accepted child types
- Layout restrictions

Hierarchy is used by:

- Editors
- Docking
- Validation
- Serialization
- Drag and Drop

---

## Reflection Pipeline

The reflection is the single source of truth.

```text
TypeScript Widgets
        │
        ▼
Reflection
        │
 ┌──────┼──────────┐
 ▼      ▼          ▼

Registry
Schema
Bindings
Documentation
Inspector
Serialization
```

The runtime never generates these artifacts.

The build system does.

---

## Decorators

Widgets opt into reflection using decorators.

```ts
@Widget({
    name: "Button",
    icon: "mouse",
    category: "Input",
})
export class Button extends Widget<ButtonProps> {

}
```

Widget properties are inferred from the widget's generic type.

Additional metadata may be provided through decorators.

---

## Generated API

The build system generates host bindings automatically.

Given

```ts
button.props.text = "Hello";
```

Rust receives

```rust
button.props.set_text("Hello");
```

Given

```ts
button.behaviour.onClick(...)
```

Rust receives

```rust
button.interface.on_click(...)
```

The generated API is entirely derived from the widget reflection.

No manual bindings are written by the user.

---

## Design Principles

- Reflection is the single source of truth.
- Runtime discovery is never performed.
- Reflection is generated only at build time.
- Generated code is never edited manually.
- Every host consumes the same reflection.
- Transport is an implementation detail.
- The runtime is unaware of the host language.
