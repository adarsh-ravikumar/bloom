# Bloom Reflection Type System

The Bloom reflection system defines a language-agnostic type system.

Widgets are authored in TypeScript, but the reflection generated by the build system is independent of TypeScript.

This allows the same reflection to generate bindings for Rust, C++, Python, documentation, inspectors, serialization, and future hosts.

---

# Philosophy

TypeScript is the authoring language.

Bloom Reflection is the contract.

Every TypeScript type used in reflected widgets is normalized into the Bloom type system.

The runtime never performs type reflection.

The build system performs reflection exactly once.

---

# Design Goals

The reflection type system must be:

- deterministic
- language independent
- serializable
- easy to map to other languages
- stable across Bloom versions

It is **not** intended to represent the entirety of the TypeScript type system.

Only the subset required by Bloom widgets is supported.

---

# Type Categories

Every reflected type belongs to one of the following categories.

```text
Type

├── Primitive
├── Enum
├── Array
├── Tuple
├── Object
├── Reference
└── Optional
```

---

# Primitive

Primitive types represent fundamental values.

Supported primitives are:

```text
String
Number
Boolean
Null
```

Examples:

```ts
text: string

count: number

enabled: boolean
```

---

# Enum

Enums represent one of several predefined values.

Enums may originate from:

TypeScript enums

```ts
enum ButtonStyle {
    Filled,
    Outline,
    Flat
}
```

or string literal unions

```ts
type ButtonSize =
    | "small"
    | "medium"
    | "large";
```

Both normalize to the same reflection representation.

---

# Array

Arrays represent ordered collections.

```ts
numbers: number[]

buttons: Button[]
```

Reflection stores the element type.

Nested arrays are supported.

---

# Tuple

Tuples represent fixed-length collections.

```ts
position: [number, number]

padding: [number, number, number, number]
```

Reflection stores the type of each element.

---

# Object

Objects represent structured values.

Objects may be declared inline

```ts
bounds: {
    width: number;
    height: number;
}
```

or through interfaces.

Objects recursively contain reflected types.

---

# Reference

References point to another reflected type.

Examples include:

```ts
Color

Vec2

Rect

Widget
```

Reflection stores only the referenced type name.

References allow custom types without modifying the reflection system.

---

# Optional

Optional represents the absence of a value.

Example:

```ts
tooltip?: string
```

Reflection represents this as

```text
Optional
    ↓
String
```

Optional is considered a type, not metadata.

---

# Constraints

Every reflected type may define constraints.

Constraints provide semantic information.

Examples include:

```text
Range

Step

Length

Pattern

Default

Display Name

Description
```

Constraints are consumed by:

- inspectors
- validation
- serialization
- generated bindings
- documentation

Constraints do not affect runtime behavior.

```ts

```

# Unsupported TypeScript Features

Bloom intentionally does not support every TypeScript feature.

The following are rejected during reflection.

```text
Conditional Types

Mapped Types

Indexed Access Types

Infer

Template Literal Types

Arbitrary Generics

Function Types

Intersection Types

Recursive Type Aliases

Unknown

Any
```

If one of these types appears inside a reflected interface, the build fails with a diagnostic.

---

# Normalization

Different TypeScript syntax may normalize to the same reflection.

Example

```ts
enum Theme {
    Dark,
    Light
}
```

and

```ts
type Theme =
    | "Dark"
    | "Light";
```

produce identical reflected enums.

Reflection represents semantics rather than syntax.

---

# Language Independence

The reflection type system is the only type system understood by generators.

Generators never inspect TypeScript.

Instead, they consume normalized reflection.

```text
TypeScript
        │
        ▼
Reflection
        │
 ┌──────┼───────────┐
 ▼      ▼           ▼

Rust   Inspector   Serialization
```

This guarantees every generated artifact is derived from the same contract.

---

# Design Principles

- Reflection is language independent.
- Reflection represents semantics, not syntax.
- Every supported type has a deterministic representation.
- Unsupported language features fail at build time.
- Reflection is the single source of truth for every generator.
